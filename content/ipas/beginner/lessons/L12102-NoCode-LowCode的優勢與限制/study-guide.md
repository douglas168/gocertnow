## 3. 核心概念

### 3.1 核心優勢一：開發速度（Development Speed）🔥

No Code / Low Code 最被強調的優勢就是**開發速度的飛躍式提升**。Forrester 研究顯示，使用 LCNC 平台的開發速度比傳統全程式碼開發（Full-Code Development）快 **20 倍**。同時，多數企業領導者認為 LCNC 顯著加快了應用交付（Forrester 調查）。

這種速度優勢來自三個層面：

```
傳統開發 vs LCNC 開發時程對比

  傳統開發（Full-Code）
  ┌──────────────────────────────────────────────────────────┐
  │ 需求分析 → 架構設計 → 編碼 → 測試 → 部署  ≈ 3–6 個月     │
  └──────────────────────────────────────────────────────────┘

  LCNC 開發
  ┌────────────────────────────────────────┐
  │ 需求分析 → 拖放設定 → 測試驗證 → 部署   │  ≈ 數天至數週
  └────────────────────────────────────────┘

  速度來源：
  ① 元件複用（Reusable Components）—— 不用從零開始
  ② 自動化測試與部署 —— 平台內建
  ③ 即時預覽（Live Preview）—— 改了馬上看到
```

🗣️ 白話說明：就像你要辦大學社團的活動報名——傳統做法是找資工系的同學從零寫一個報名網站，可能要兩個禮拜。但用 Google AppSheet 或 Airtable，你午餐時間就能從 Google Sheets 變出一個有表單、有通知的報名 App。這就是 20 倍速差距的真實感。

**快速原型（Rapid Prototyping）** 是這個優勢最典型的應用場景：用 LCNC 快速做出 MVP（Minimum Viable Product, 最小可行產品），先驗證市場需求，再決定是否投入完整開發資源。

### 3.2 核心優勢二：成本節省（Cost Reduction）🔥

LCNC 的成本優勢不只是「少請工程師」，而是體現在整個開發生命週期（Development Lifecycle）：

| 成本面向 | 傳統開發 | LCNC 開發 |
|---------|---------|----------|
| 人力成本 | 需聘專業工程師團隊 | 業務人員可自行開發 |
| 時間成本 | 3–6 個月 | 數天至數週 |
| 維護成本 | 團隊自行維護伺服器與程式碼 | 平台負責底層基礎架構 |
| 試錯成本 | 高（失敗 = 大量沉沒成本） | 低（快速迭代、快速放棄） |

🗣️ 白話說明：台灣一個 Junior 工程師月薪 4–5 萬，Senior 8–10 萬。如果一個小型企業要做一個內部的請假系統，傳統做法要請人寫三個月，光薪資就 12–30 萬。但用 Power Apps，一個懂 Excel 的人資同事花一週就能做出來，只需要平台訂閱費。

**台灣在地脈絡**：數位時代報導指出，**43% 的台灣企業領導者**認為數位轉型最大的障礙是技術人才不足。LCNC 正好填補了這個缺口——不需要搶破頭在 104 上找工程師，讓現有員工就能動手做。

### 3.3 核心優勢三：公民開發者賦能（Citizen Developer Empowerment）🔥

承接 L12101 介紹的公民開發者（Citizen Developer）概念，本課進一步討論其**實際效益**：

- Gartner 預測：2026 年 **80%** 的低程式碼工具用戶來自正式 IT 部門以外 🔥
- 效益：**業務部門自主解決需求**，不需要等 IT 排程（Development Backlog）
- IT 部門角色轉變：從「什麼都要自己寫」→ 變成「制定規範、監督治理」

```
公民開發者的價值鏈

  業務需求產生          傳統路徑                    LCNC 路徑
  ┌──────────┐
  │ 業務部門   │──→ 提需求給 IT ──→ 排隊 ──→ 3個月後交付
  │ 發現問題   │
  │          │──→ 用 NC/LC 工具 ──→ 自己做 ──→ 1週搞定 ✅
  └──────────┘
```

🗣️ 白話說明：想像你在公司業務部，每次要做一個客戶追蹤表都要提 IT 需求單，IT 部門排到你可能是兩個月後。有了 LCNC，你自己用 Airtable 花一個下午就做好了，不用等、不用求人。

### 3.4 核心優勢四：AI 民主化（AI Democratization）🔥

AI 民主化（AI Democratization）是指**降低 AI 技術的使用與開發門檻**，讓非技術專業人員也能建置和使用 AI 應用。LCNC 是實現 AI 民主化的關鍵推手之一。

具體體現：
- **Google AppSheet + AI**：直接在 No Code 環境中加入 AI 預測功能
- **Microsoft Power Platform + Copilot**：在 Low Code 環境中用自然語言生成應用
- **生成式 AI + LCNC 整合**：用 AI 自動產生介面設計、行銷文案、報表分析

🗣️ 白話說明：以前要做一個「能根據客戶資料自動推薦產品」的系統，你需要請資料科學家寫 Python 訓練模型，再請工程師把模型包成 API，前後端串接⋯⋯現在用 Power Apps + AI Builder，行銷部門的人自己拖一拖就能建一個有 AI 推薦功能的 App。AI 不再是工程師的專利。

> ⚠️ 注意：LCNC 讓 AI 更容易使用，但**不代表不需要人類監督**。AI 模型的偏差（Bias）、資料品質等問題仍需人工把關。這是考試常見的「過度樂觀」陷阱。

### 3.5 核心限制一：供應商鎖定（Vendor Lock-in）🔥🔥

供應商鎖定（Vendor Lock-in）是 LCNC 最被討論的風險，也是**考試最愛考的限制**。

定義：當企業深度依賴某個 LCNC 平台後，**無法輕易將應用遷移到其他平台或自建環境**。多數平台（尤其 No Code）無法匯出原始碼；部分 Low Code 平台雖允許匯出，產出的程式碼仍高度依賴原廠環境套件，遷移成本依然很高。所有的邏輯、資料結構、介面設計都綁在該平台的專有格式（Proprietary Format）中。

```
Vendor Lock-in 的困境

  Day 1：開心上線                      Day 365：想換平台
  ┌──────────┐                       ┌──────────┐
  │ 選了平台 A │                       │ 平台 A 漲價 │
  │ 快速開發 ✅ │                       │ 想搬到 B   │
  │ 超級方便   │                       │           │
  └──────────┘                       └──────────┘
                                            │
                                     無法匯出原始碼
                                            │
                                            ▼
                                     ┌──────────┐
                                     │ 遷移 = 重寫 │
                                     │ 💸 巨額成本  │
                                     └──────────┘
```

🗣️ 白話說明：就像你把所有照片都存在某個雲端相簿裡，用了三年後發現它要漲價，想搬家卻發現照片格式是它自己的專利格式、匯不出原檔。LCNC 的 Vendor Lock-in 就是這種感覺——進去容易，出來很痛。

**為什麼這是最重要的限制？** 因為它直接影響企業的**長期策略自主性**。一旦被鎖定，企業對技術選型（Technology Selection）的談判籌碼就大幅降低。

### 3.6 核心限制二：影子 IT（Shadow IT）🔥🔥

影子 IT（Shadow IT）是指**組織中未經 IT 部門知情或核可的技術系統與應用**。LCNC 工具因為門檻低，特別容易催生影子 IT。

```
影子 IT 的形成路徑

  業務部門覺得好用
        │
        ▼
  自己註冊 NC/LC 帳號
        │
        ▼
  建了一堆內部工具
        │
        ▼
  IT 部門完全不知道
        │
        ▼
  ┌─────────────────────────────┐
  │ 風險一：資料安全漏洞          │
  │ 風險二：不符合法規合規（GDPR 等）│
  │ 風險三：系統間無法整合          │
  │ 風險四：員工離職後無人維護       │
  └─────────────────────────────┘
```

🗣️ 白話說明：想像大學社團——社長自己辦了一個 Google Form 收社費，把大家的銀行帳號存在自己的 Google Sheets 裡。學校的資安中心完全不知道這件事。如果 Sheets 權限沒設好，個資就外洩了。公司裡的影子 IT 就是這個情況的放大版。

**影子 IT vs 公民開發者的區別（常考）：**
- **公民開發者**：使用**組織核可**的工具、在**IT 治理框架**下開發 → 正面
- **影子 IT**：使用**未經核可**的工具、**IT 不知情** → 負面風險

---

### 3.7 核心限制三：擴展性瓶頸（Scalability Bottleneck）

當應用從簡單的內部工具成長為需要處理大量使用者或複雜商業邏輯（Complex Business Logic）的系統時，LCNC 平台可能遇到擴展性瓶頸（Scalability Bottleneck）。

主要瓶頸：
1. **複雜邏輯限制**：NC/LC 平台的視覺化流程設計器難以表達深層巢狀邏輯（Nested Logic）
2. **高併發限制**：當同時上線人數暴增，平台效能可能無法承受
3. **整合深度限制**：No Code 工具（如 Zapier）的 API 串接能力有限，複雜邏輯做不到

🗣️ 白話說明：用 Canva 做一張 IG 貼文沒問題，但如果你要做一整套電影特效，Canva 就力不從心了。LCNC 也一樣——做一個報名表單很棒，但要做像蝦皮那樣同時處理百萬人搶購的系統，就超出它的能力範圍了。

> 💡 考試怎麼考：「No Code 工具如 Zapier 的應用限制為？」正答：**複雜邏輯有限制**（不是「不支援 API」——Zapier 其實支援 API 串接，只是複雜邏輯做不到）。

---

### 3.8 核心限制四：技術債（Technical Debt）

技術債（Technical Debt）是指為了**短期快速交付而犧牲長期程式品質**所累積的問題。LCNC 因為強調「快」，特別容易產生技術債。

形成路徑：
1. 用 NC/LC 快速做出 MVP → 上線成功
2. 需求持續增加 → 在原有基礎上不斷疊加功能
3. 從未重構（Refactoring）→ 系統越來越混亂
4. 最終：修一個 bug 會產生三個新 bug → 維護成本飆升

> 💡 **補充限制——版本控制不足**：多數 LCNC 平台缺乏類似 Git 的分支／合併（Branching/Merging）機制，多人同時修改同一應用時容易互相覆蓋，團隊協作的困難度隨人數增加而急劇上升。

🗣️ 白話說明：就像你的大學報告——第一版是趕出來的，之後老師改了需求你就在上面東補西補，到第五版已經亂七八糟、自己都看不懂。LCNC 的技術債就是這種「一直東拼西湊卻從不整理」的後果。

### 3.9 各場域應用情況（Domain Applications）🔥

考試可能以台灣產業情境出題。以下是 LCNC 在各領域的典型應用：

| 場域 | 典型 LCNC 應用 | 常用工具類型 |
|------|---------------|-------------|
| **金融保險** | 理賠流程自動化、KYC 表單 | Low Code（合規需求高） |
| **醫療** | 掛號系統、排班管理、病歷查詢介面 | Low Code（資安需求高） |
| **零售電商** | 庫存管理、行銷活動自動化、LINE 推播 | No Code + Low Code |
| **教育** | 報名表單、成績追蹤、家長通知 | No Code |
| **製造業** | 產線數據監控、良率報表 | Low Code |
| **新創 / SME** | MVP 快速驗證、內部工具 | No Code |

**台灣在地案例**：蘋果供應鏈上的台灣製造業者，已使用 LCNC 搭配大數據，讓**產線人員**（非工程師）自行匯入機台數據、監控良率。這正是公民開發者 + LCNC 在製造場域的實踐。

> ⚠️ **場域選擇原則**：需要高度安全性、合規性的場域（金融、醫療）傾向選擇 **Low Code**（因為有 IT 監督與治理機制）；需求簡單、快速迭代的場域（教育、新創）傾向 **No Code**。
>
> ⚠️ **資料落地限制**：受法規監管的產業（金融、醫療）可能有資料駐留（Data Residency）要求——全球性 SaaS LCNC 平台未必在台灣設有伺服器，這在合規評估時是需要考量的風險。

### 3.10 Low Code 平台的模型概念（Model）🔥

114 年第四次初級考試已出現此概念。在 Low Code 平台中，**模型（Model）** 是用來**抽象描述資料結構（Data Structure）、業務流程（Business Process）與介面邏輯（UI Logic）的核心元件**。

```
Low Code 平台的「模型」概念

  模型（Model）
  ┌─────────────────────────────────┐
  │                                 │
  │  📊 資料模型（Data Model）         │  ← 描述資料欄位與關聯
  │  🔄 流程模型（Process Model）      │  ← 描述業務流程步驟
  │  🖥️ 介面模型（UI Model）          │  ← 描述畫面佈局與互動
  │                                 │
  └─────────────────────────────────┘
       ↓ 平台自動轉換
  ┌─────────────────────────────────┐
  │  可執行的應用程式（Executable App） │
  └─────────────────────────────────┘
```

🗣️ 白話說明：資料模型像定義 Excel 欄位，流程模型像畫心智圖決定步驟，介面模型像用 Canva 排版。三者結合才是 Low Code 的模型。

> 📌 **教學提示**：此三分法（資料模型／流程模型／介面模型）為教學架構，幫助理解模型概念的三個面向，考試以整體定義為主。

> ⚠️ **重要陷阱**：此處的「模型（Model）」是 Low Code 平台的抽象設計概念，**不是機器學習模型（Machine Learning Model）**。考試會利用這個同名詞來混淆！

### 3.11 LCNC + 生成式 AI 整合（新趨勢考點）🔥

這是 114 年第四次考試已出現的新趨勢考點。LCNC 平台正在與生成式 AI（Generative AI）整合，產生新的應用模式：

| 整合方式 | 說明 | 範例 |
|---------|------|------|
| **AI 生成介面** | 用自然語言描述需求，AI 自動產生 App 介面 | Power Apps Copilot |
| **AI 輔助開發** | AI 建議工作流程配置、自動補全邏輯 | Zapier AI Actions |
| **AI 內容產出** | LCNC 平台內嵌生成式 AI 做行銷文案、報表摘要 | AppSheet + Gemini |

🗣️ 白話說明：以前用 LCNC 你要自己拖拉元件來設計畫面。現在你可以直接跟 AI 說「幫我做一個客戶管理 App，要有聯絡人清單和跟進記錄」，AI 就幫你把 App 的雛形生出來，你再微調。LCNC + AI = 速度再乘以速度。

> 📌 **考題型態**：「設計團隊需結合 NC/LC 平台與生成式 AI，快速完成介面生成和行銷內容自動產出」——此類情境整合題需要同時理解 LCNC 優勢和 AI 能力。

## 4. 比較表（易混淆概念）

### 4.1 No Code 的優勢 vs Low Code 的優勢 🔥

| 面向 | No Code 的優勢 | Low Code 的優勢 |
|------|---------------|----------------|
| **門檻** | 最低——零程式碼 | 低～中——需少量程式知識 |
| **速度** | 最快——小時到天 | 快——天到週 |
| **適合使用者** | 非技術業務人員 | 開發者 / 公民開發者 |
| **適合場景** | 簡單工具、表單、自動化 | 企業級應用、複雜整合 |
| **客製化** | 低（受限模板） | 高（可寫程式碼擴展） |
| **IT 治理** | 較難管控 | 較易納入治理框架 |

### 4.2 No Code vs Low Code vs Traditional 三方比較

| 面向 | No Code | Low Code | Traditional（全程式碼） |
|------|---------|----------|----------------------|
| **速度** | 最快（小時～天） | 快（天～週） | 慢（月～季） |
| **客製化** | 低（受限模板） | 中高（可寫程式碼擴展） | 最高（完全自由） |
| **IT 治理** | 較難管控 | 可納入治理框架 | 完全由 IT 掌控 |
| **擴展性** | 低（簡單場景） | 中（企業級應用） | 高（無上限） |
| **適用場景** | 表單、自動化、內部小工具 | 企業應用、流程系統 | 高併發、複雜演算法、核心系統 |

### 4.3 Vendor Lock-in vs Shadow IT（最常考的兩大風險）🔥🔥

| 面向 | Vendor Lock-in（供應商鎖定） | Shadow IT（影子 IT） |
|------|---------------------------|---------------------|
| **定義** | 深度依賴平台，無法遷移 | 未經 IT 核可的系統 |
| **風險本質** | 失去技術選擇自主權 | 失去安全與合規控制 |
| **誰造成的** | 平台本身的封閉設計 | 使用者自行繞過 IT |
| **影響對象** | 企業的長期策略 | 企業的資安與合規 |
| **解方** | 評估平台開放性、匯出能力 | 建立治理框架、核可工具清單 |
| **考題關鍵字** | 「無法遷移」「換平台要重寫」 | 「IT 不知情」「未經核可」 |

### 4.4 公民開發者 vs 影子 IT（概念邊界）

| 面向 | 公民開發者（Citizen Developer） | 影子 IT（Shadow IT） |
|------|------------------------------|---------------------|
| **工具** | 組織**核可**的 NC/LC 平台 | **未經核可**的工具 |
| **IT 知情** | 是——在 IT 治理框架下 | 否——IT 完全不知道 |
| **性質** | 正面——企業鼓勵的趨勢 | 負面——需要防範的風險 |
| **考試立場** | 題目呈正面語境 | 題目呈風險語境 |

### 4.5 Low Code「模型」vs 機器學習「模型」

| 面向 | Low Code 模型（Model） | ML 模型（ML Model） |
|------|----------------------|---------------------|
| **定義** | 抽象描述資料結構、流程與介面 | 從數據中學習的數學函數 |
| **產出** | 可執行的應用程式 | 預測或分類結果 |
| **使用者** | 公民開發者 / 開發者 | 資料科學家 / ML 工程師 |
| **所屬領域** | 軟體開發 | 人工智慧 / 機器學習 |

## 5. 口訣 / Mnemonics

### 口訣一：LCNC 優勢三角

> **「快省民」**
>
> - **快** → 開發速度（快 20 倍）
> - **省** → 成本節省（少請工程師）
> - **民** → 公民開發者賦能 + AI 民主化

### 口訣二：四大風險（V-S-S-T）

> **「鎖影脹債」**（供應商「鎖」住你、「影」子 IT、膨「脹」不了、欠「債」要還）
>
> - **鎖** → Vendor Lock-in（供應商鎖定）
> - **影** → Shadow IT（影子 IT）
> - **脹** → Scalability Bottleneck（擴展性瓶頸，想膨脹卻脹不了）
> - **債** → Technical Debt（技術債）

### 口訣三：場域選擇原則

> **「安全選 Low，簡單選 No」**
>
> - 需要高安全性 / 合規性（金融、醫療）→ Low Code
> - 需求簡單 / 快速迭代（教育、新創）→ No Code

### 口訣四：影子 IT vs 公民開發者

> **「核可是公民，偷做是影子」**
>
> - 組織核可 → 公民開發者（正面）
> - 未經核可 → 影子 IT（負面）

## 6. 考試陷阱

### 陷阱一：Zapier 不支援 API 🔥🔥

❌ 陷阱：看到「Zapier 的限制」就選「不支援 API」

✅ 正解：Zapier **支援 API 串接**（它就是靠 API 串接各種服務的）。Zapier 真正的限制是**複雜邏輯有限制**——它的 Trigger-Action 模式難以處理深層巢狀邏輯或複雜的條件分支。這是 114 年考古題的考點，務必記住。

---

### 陷阱二：LCNC 能取代所有傳統開發

❌ 陷阱：以為 NC/LC 發展到最後可以完全取代傳統程式開發

✅ 正解：LCNC 擅長的是中小型、標準化的應用場景。**高度客製化、高併發、複雜演算法**的需求仍然需要傳統全程式碼開發。兩者是互補關係（Complementary），不是替代關係（Replacement）。

---

### 陷阱三：Low Code 平台的「模型」= ML 模型 🔥

❌ 陷阱：看到「模型（Model）」就聯想到機器學習模型

✅ 正解：Low Code 平台中的「模型」是指**抽象描述資料結構、業務流程與介面邏輯的設計元件**，跟 ML 的模型（訓練出來做預測的數學函數）完全不同。114 年第四次考試已考過此概念。

---

### 陷阱四：公民開發者越多越好，不需要管理

❌ 陷阱：以為鼓勵公民開發者 = 放任大家自己做，IT 部門不用管

✅ 正解：公民開發者的正向價值**建立在 IT 治理框架之上**。沒有治理的公民開發 = 影子 IT = 風險。企業應該做的是「提供核可工具清單 + 制定開發規範 + 定期審查」，而不是完全放手。

---

### 陷阱五：LCNC + AI = 不需要人類參與

❌ 陷阱：以為 LCNC 加上生成式 AI 後，整個開發流程可以全自動、零人工

✅ 正解：AI 可以加速介面生成和內容產出，但**業務需求判斷、資料品質把關、AI 偏差監督**仍需人類參與。「AI 民主化」降低的是技術門檻，不是消除人的角色。

---

### 陷阱六：Vendor Lock-in 只影響成本

❌ 陷阱：以為 Vendor Lock-in 的問題就是「換平台要花錢」

✅ 正解：Vendor Lock-in 影響的不只是遷移成本（Migration Cost），更包括**技術策略自主性**（Strategic Autonomy）。被鎖定後，企業在功能需求、定價談判、安全更新等方面都被動。這才是考試要你理解的深層意涵。

---

## 7. 情境題快速判斷

### 🔑 看到關鍵字 → 選這個答案

**風險辨識類（最高頻）：**
- 「無法遷移」「換平台要重寫」「被平台綁住」 → **Vendor Lock-in（供應商鎖定）**
- 「IT 不知情」「未經核可」「員工自行建置」 → **Shadow IT（影子 IT）**
- 「系統變複雜後效能下降」「高併發撐不住」 → **擴展性瓶頸（Scalability Bottleneck）**
- 「快速上線後維護困難」「疊床架屋越來越亂」 → **技術債（Technical Debt）**

**優勢辨識類：**
- 「快速原型」「MVP」「開發時間大幅縮短」 → **開發速度**
- 「減少工程師依賴」「人才缺口」 → **成本節省**
- 「非 IT 人員自行開發」「業務部門自主」 → **公民開發者賦能**
- 「非技術人員建置 AI 應用」「降低 AI 門檻」 → **AI 民主化**

**場域應用類：**
- 「理賠流程」「KYC」 → 金融保險 → 傾向 **Low Code**
- 「排班系統」「病歷查詢」 → 醫療 → 傾向 **Low Code**
- 「報名表單」「成績追蹤」 → 教育 → 傾向 **No Code**
- 「產線監控」「良率報表」 → 製造業 → 傾向 **Low Code**
- 「MVP 驗證」「內部小工具」 → 新創 → 傾向 **No Code**

**工具限制類：**
- 「Zapier 的限制」 → **複雜邏輯有限制**（不是不支援 API！）
- 「No Code 做不到的」 → **高度客製化、複雜邏輯、高併發**
- 「模型（Model）在 LC 平台的意思」 → **抽象描述資料結構、流程與介面**（不是 ML 模型）

### 🔑 情境題速查表

| 情境描述 | 正確答案 | 關鍵判斷 |
|---------|---------|---------|
| 企業用了三年的 NC 平台宣布大幅漲價，想搬走卻發現做不到 | Vendor Lock-in | 無法遷移 |
| 行銷部自己註冊了一堆雲端工具，IT 主管完全不知道 | Shadow IT | IT 不知情 + 未核可 |
| 一開始用 Zapier 做的自動化流程，需求越來越複雜後開始出錯 | 擴展性瓶頸 | 複雜邏輯超出工具能力 |
| 去年用 NC 快速上線的系統，現在每次改功能都會壞掉其他地方 | 技術債 | 未重構 + 維護困難 |
| 醫院想做一個掛號系統，需要符合個資法規 | Low Code | 高安全 + 合規需求 |
| 大學社團想做一個活動報名 App | No Code | 非技術人員 + 簡單需求 |
| 設計團隊想用 AI 自動生成 App 介面 + 行銷文案 | LCNC + 生成式 AI 整合 | AI 輔助 + 快速開發 |
| 公司想讓業務部門自己做客戶管理工具，但要在 IT 監管下 | 公民開發者模式 | 核可工具 + IT 治理 |
| 非技術主管想用 AI 做銷售預測 App | AI 民主化 + LCNC | 非技術 + AI 應用 |

### 🔑 四步驟風險判斷法

遇到「某企業使用 NC/LC 後遇到問題，最可能的原因是？」這類題目時：

1. **看能不能走**：被平台綁住走不了？→ Vendor Lock-in
2. **看 IT 知不知道**：IT 部門不知情？→ Shadow IT
3. **看系統長大了沒**：需求變複雜撐不住？→ 擴展性瓶頸
4. **看有沒有整理**：一直疊功能沒重構？→ 技術債

---

> ✅ 本課重點回顧：LCNC 優勢記「快省民」（速度快、成本省、公民開發者 + AI 民主化）；限制記「鎖影脹債」（Vendor Lock-in、Shadow IT、擴展性瓶頸、技術債）；Zapier 限制是「複雜邏輯」不是「不支援 API」；Low Code 的「模型」是設計元件不是 ML 模型；安全場域選 Low Code、簡單場域選 No Code。搞定這些加上 L12101 的基礎，L121 整個單元的分數就穩了。
