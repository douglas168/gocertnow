## Section 3: 核心概念詳解

### 3.1 🎯 目標設置（Goal Setting） 🔥🔥

#### 3.1.1 SMART 原則（SMART Criteria）

目標不是寫「我們要變強」，而是要能被老闆、財務、法遵、使用者**同時看懂且驗收**。業界最通用的就是 **SMART 原則（SMART Criteria）**：

| 英文 | 中文 | 意思 |
|---|---|---|
| **S**pecific | 具體 | 誰、做什麼、對誰、在哪個流程 |
| **M**easurable | 可量化 | 有數字、有指標、能被驗收 |
| **A**chievable | 可達成 | 資源、時間、技術上做得到 |
| **R**elevant | 相關 | 對齊公司策略、解決真問題 |
| **T**ime-bound | 有期限 | 幾個月內達成、里程碑清楚 |

🗣️ **白話說明**：想像你跟同學組報告，隊長說「我們要做得很棒」——這就不是 SMART。換成「下週三前交 20 頁、有 5 張圖、比對三家競品」——這才是能動手的目標。AI 專案也一樣，「提升客服品質」太空泛，「3 個月內把客服首次回應時間從 5 分鐘降到 2 分鐘」才是 SMART。

🔥🔥 **考試焦點**：題目會給你四個目標描述，要你挑哪個「最符合 SMART」。選有**數字 + 期限 + 具體流程**的那個。

🚫 反例：某製造業若做「AI 公司口號產生器」，技術上可行但與業務無關 → Relevant 不合格。同樣預算改做「工安手冊問答機器人」，直接對應生產安全痛點 → Relevant 合格。Relevant 檢核的是「與企業策略／部門痛點對齊」，而不是「技術上做不做得到」。

#### 3.1.2 可量化 KPI（Key Performance Indicators）

AI 導入常用 KPI 範例：

```
客服場景：
  • 首次回應時間：5 min → 2 min（降 60%）
  • AI 一次解決率：從 30% 提升至 55%
  • 客服專員處理量：每日 60 通 → 90 通

內容生成場景：
  • 社群貼文產出時間：每則 40 min → 10 min
  • 錯字／品牌違規率：< 2%
  • 採用率（採用 AI 初稿直接發）：> 70%

法遵/文件場景：
  • 合約關鍵條款抽取準確率：≥ 95%
  • 人工審閱時間減少：40%
  • 遺漏率：< 1%
```

🗣️ **白話說明**：KPI 就像健身房的「體脂、深蹲重量、3 公里時間」——不是「我覺得變強」，是**可以站上體重計的數字**。寫 AI 專案 KPI 也是一樣，沒有數字，三個月後就沒辦法開檢討會。

#### 3.1.3 對齊企業策略

🔥 目標不是「IT 部門自嗨」，要向上對齊公司今年的營運主軸（例：降本、提升客戶滿意度、擴大市場）。否則就算技術做成功，經營層不買單，明年預算就沒了。

（OKR 屬中級目標框架，初級不需掌握定義；SMART 即可滿足考試需求。）

### 3.2 💰 資源分配（Resource Allocation） 🔥

#### 3.2.1 資源四維度

AI 專案不是「買個訂閱就能動」。要同時盤點**四個維度**：

```
        ┌────────────┐
        │ 👥 人力     │  PM、領域專家、IT、資料標註、法遵/資安、訓練
        └────────────┘
              │
  ┌───────────┼───────────┐
  │           │           │
┌─▼────┐  ┌───▼────┐  ┌───▼────┐
│💵 預算│  │📦 資料 │  │⏰ 時間 │
│      │  │        │  │        │
│API費 │  │ 品質   │  │ 里程碑 │
│授權  │  │ 數量   │  │ 排程   │
│訓練  │  │ 隱私   │  │ 回饋   │
│維運  │  │ 標註   │  │ 迭代   │
└──────┘  └────────┘  └────────┘
```

🗣️ **白話說明**：想像你要在宿舍辦一場 30 人的聖誕派對。光有錢（預算）不夠，還要有場地（資料/環境）、有人佈置（人力）、有時間籌備（時間）。AI 專案一樣，四樣少一個就翻車。

#### 3.2.2 四維度詳細說明

**👥 人力**：AI 導入**不是 IT 單一部門**的事。典型跨部門編組：
- **PM（專案管理）**：對齊目標、控時程。
- **領域專家**：業務/客服/法務——AI 才知道什麼是對的答案。
- **IT / 資料工程**：串接系統、整備資料。
- **資料標註人員**：把資料打上正確答案給 AI 學。
- **法遵 / 資安**：看個資、看合規。
- **訓練與溝通**：教使用者怎麼用。
- **會寫 Prompt 的領域專家（Prompt Engineering SME）**：2025–2026 台灣市場稀缺資源。純 PM 或純 IT 都不夠——需要懂業務流程又能寫出穩定 prompt 的人，且此能力無法短期外包，需在規劃階段就盤點內部人選或培訓預算。

🗣️ 就像系上辦迎新，不可能只叫活動股做。總務要出錢、學藝出美宣、公關找贊助——缺一個就要加班到半夜。

**💵 預算**：不要只看第一個月的訂閱費。要看 TCO（Total Cost of Ownership 總擁有成本）概念——包含 API token 使用費、授權費、導入顧問、員工訓練、後續維運。（中級會教詳細 TCO 公式，初級只需知道「成本不只訂閱費」。）

💸 **Token 成本漂移（Cost Drift）**：GenAI 與傳統軟體固定授權費不同，成本**隨使用量線性上升**。Pilot 成功但使用量暴增時，若 Scale 階段仍沿用原預算會破表。規劃時應估算每月尖峰 token 用量 × 單價 × 安全緩衝（建議 1.5–2 倍）。

**📦 資料**🔥：**最常被低估的資源維度**。台灣產業調查顯示僅 **5–10% 企業具備可用於 AI 的資料**（AIF 2025 台灣產業 AI 化大調查）。資料要看：
- **品質**：有沒有錯字、過時、缺漏。
- **數量**：量夠不夠訓練／驗證。
- **隱私**：個資、病歷、財務資料能不能用。
- **標註**：有沒有人工打上正確答案。

🗣️ 這就像你要開烘焙店，光有食譜（模型）、有錢買烤箱（工具）沒用，**麵粉雞蛋品質爛（資料爛），做出來的蛋糕一定難吃**。

⚖️ **台灣個資法（PDPA）規劃要點**：規劃 GenAI 專案時，若使用真實用戶資料，**必須在 POC 啟動前**完成以下之一：（1）去識別化（**Anonymization**）、（2）取得使用者**知情同意（Informed Consent）**、或（3）改用合成／公開資料。不能等 Scale 階段才補法遵，否則專案要整體退回。（此為初級規劃意識層面，詳細合規流程屬 L11203 資料隱私與安全。）

**⏰ 時間**：要有清楚里程碑（Milestone），而且**留回饋與修正的時間**。AI 專案常見錯誤是「把時程排到週五上線，沒有留測試與訓練週」。

### 3.3 🚀 導入策略（Adoption Strategy） 🔥🔥

#### 3.3.1 POC → Pilot → Scale 三階段

業界共識的導入節奏：

```
┌──────────┐    ┌──────────┐    ┌──────────┐
│ POC 概念 │───▶│ Pilot 試 │───▶│ Scale 擴 │
│   驗證    │    │   點     │    │   大推展  │
├──────────┤    ├──────────┤    ├──────────┤
│ 技術可行 │    │ 商業價值 │    │ 全面上線 │
│   嗎?    │    │ 真的有?  │    │ + SOP    │
│ 小規模   │    │ 1-2 部門 │    │ + 監控    │
│ 短週期   │    │ 真實用戶 │    │ + 訓練    │
└──────────┘    └──────────┘    └──────────┘
    2-4 週           1-3 月          持續
```

🔥🔥 考試愛考「**三階段的順序、每階段的主要目的**」：

| 階段 | 英文 | 目的 | 規模 |
|---|---|---|---|
| **POC** | Proof of Concept | 驗證技術可行性 | 最小範圍、幾個人 |
| **Pilot** | Pilot | 驗證商業價值 | 1–2 個單位、真實使用者 |
| **Scale** | Scale / Rollout | 全面上線 | 跨部門、建 SOP、監控 |

> **Scale（擴大推展）**：進入正式生產環境並建立標準化流程（Production Standardization），不必然等於全公司所有人都使用；重點在於流程、權限、監控、成本控管已標準化。

🗣️ **白話說明**：就像你想開手搖飲品牌。
- **POC**：先自己在家煮，看茶葉跟糖水比例行不行。
- **Pilot**：在學校門口擺一天快閃，真的有人買嗎？能不能賺？
- **Scale**：這時才租店面、做品牌、開分店。

如果你家裡煮一杯好喝（POC 成功），就直接租三間店面（Scale），**十之八九會倒**——因為還沒驗證真實客戶買不買單、流程順不順（Pilot 沒做）。

#### 3.3.2 退出準則（Exit Criteria）

🔥 每個階段**進下一階段前**，要先定義「什麼條件算過關」。例：
- POC 退出準則：答案準確率 ≥ 80%、回應時間 < 3 秒。
- Pilot 退出準則：使用者滿意度 ≥ 4.0 / 5、節省工時 ≥ 20%、無重大錯誤事件。

沒有退出準則，就會變「感覺還可以，那就上線吧」——然後上線後一路漏水。

退出準則的門檻須依**風險等級**設定：醫療、法律、金融等高風險場域可能需要 95%+ 的準確率與 HITL 抽查；內部 FAQ 或創意輔助類低風險場景 70–80% 即可接受。單一「80%」數字並非通用答案。

#### 3.3.3 並行運行 vs 完全替換

```
並行運行 (Parallel Run)         完全替換 (Full Replacement)
┌──────────────────┐            ┌──────────────────┐
│  舊系統 ─→ 結果A │            │                  │
│                  │            │  新 AI 系統 ─→   │
│  新 AI ─→ 結果B  │            │      結果        │
│                  │            │                  │
│  ⚖️ 比對 A vs B   │            │  (舊系統下線)    │
└──────────────────┘            └──────────────────┘
   風險低、成本高                  風險高、成本低
   推薦：金融/醫療/法遵            推薦：低風險/新建流程
```

🗣️ **白話說明**：並行運行就像你在補習班時，新老師上課你也繼續複習舊講義，兩邊對照確定沒漏掉再放手。完全替換就像直接把舊講義丟掉——萬一新老師教錯你就來不及了。

🗣️ 白話說明：GenAI 的並行運行比傳統系統難——兩個模型回答同一個 prompt 不會產生一模一樣的輸出（**非確定性，Non-deterministic**），所以「A 版 vs B 版哪個好」不能像比對帳務數字那樣自動比較。實務上需要「LLM-as-a-judge」或人工抽樣，使得並行運行的人力成本比想像中高。

#### 3.3.4 跨部門協作 🔥

AI 導入**絕對不是 IT 部門一個人的事**。常考正解：
- IT / 資料工程：串接、資料整備。
- 業務 / 客服：提真實需求、驗證場景。
- 法遵 / 法務：個資、合約、AI 標示合規。
- 資安：資料外洩風險、Prompt Injection 防範（概念層）。
- HR / 訓練：員工適應、SOP、操作手冊。

### 3.4 🛡️ 因應措施（Contingency Measures） 🔥🔥

AI 會出錯、會幻覺、會被攻擊、會有偏見。因應措施就是**「AI 出包時，系統還能活下去」的備案**。業界共識四件套：

```
        🛡️ 因應措施四件套
┌───────────┬───────────┬───────────┬───────────┐
│ ① HITL    │ ② Fallback│ ③ Rollback│ ④ 幻覺偵測│
│ 人類介入  │ 備援降級  │ 版本回滾  │ + 標示    │
├───────────┼───────────┼───────────┼───────────┤
│ AI 決策   │ AI 掛了   │ 上版本壞  │ AI 說錯?  │
│ 誰按確認  │ 切回規則  │ 退回舊版  │ 事實查核  │
│ 誰負責任  │ 或人工    │ 不能變壞  │ 標 AI生成 │
└───────────┴───────────┴───────────┴───────────┘
```

#### 3.4.1 HITL 人類介入（Human-in-the-Loop） 🔥🔥🔥

**定義**：AI 產出**不直接發給終端使用者 / 不直接執行決策**，一定要由**人類在迴圈裡覆核或簽核**。

**必備情境（高風險）**：
- 🏥 醫療：診斷建議、病歷摘要、藥物交互作用。
- 💰 金融：授信、投資建議、反詐欺。
- ⚖️ 法律 / 勞權：解僱、薪資、合約建議。
- 🏛️ 公部門：公文決定、福利核發。

🗣️ **白話說明**：AI 就像實習生，報表可以叫他先做，但**最終簽名的一定要是主管**。如果今天銀行 AI 說「拒絕你的貸款」就直接寄信，你根本沒地方申訴——所以**金融法遵強制要有人審核**。這就是 HITL。

🔥🔥 **考試套路**：題目只要提到「醫療、金融、勞權、法律、高風險決策、問責」——幾乎一定要選 **HITL / 人類介入 / 人工覆核**。

```
HITL 決策流程
  ┌─────────┐
  │ AI 產出 │
  └────┬────┘
       │
  ┌────▼─────┐    高     ┌─────────────┐
  │ 風險分級 │─────────▶│ 人工必審核  │──▶ 輸出
  └────┬─────┘           └─────────────┘
       │ 低
       ▼
  ┌─────────────┐
  │ 人工抽查   │──────────────────────▶ 輸出
  │ + 事後稽核 │
  └─────────────┘
```

#### 3.4.2 Fallback 備援降級

AI 當機、API 超時、回覆品質掉下去時，**切換回規則引擎或人工服務**，不要讓使用者卡住。

🗣️ 就像 7-11 的 POS 壞掉，店員還是會改用計算機手開發票——**服務不能斷**。

#### 3.4.3 版本回退／回滾（Rollback，台灣 IT 業界兩種講法並用）

上了新版本結果表現比舊版差，要能**快速退回上一個穩定版本**。不能說「已經上線就回不去了」。

🗣️ 就像 iOS 更新後變超頓，你想退回上一版。有準備 Rollback 就能退，沒準備就只能硬用。

🗣️ 記憶鉤子：Fallback 像「**備用發電機**」——主服務一掛掉立刻切換，使用者感覺不到，屬即時、暫時的應變。Rollback 像「**拆掉施工不良的工程回到前版**」——發現版本上線後有結構性問題，整體退回上一個穩定版本，屬永久、版本層級的操作。

#### 3.4.4 幻覺偵測 + AI 內容標示

- **幻覺偵測（Hallucination Detection）**：AI 一本正經講假話——要有事實查核（fact-checking）、參考來源引用、信心分數門檻。
- **內容標示**：行政院 2023 指引明定，公部門使用 GenAI 產出應「**標示為 AI 生成**」並由人員覆核。屬於合規等級的因應措施。

#### 3.4.5 偏見監測（概念層）

AI 可能對特定族群（性別、年齡、地區）產生不公平結果，需持續監測。（此為中級內容，初級只需了解概念，不需會演算法或公式。）

📝 **著作權歸屬（AI 生成內容）**：規劃階段須釐清 AI 生成物的著作權歸屬——依台灣經濟部智慧財產局（TIPO）現行見解，**純由 AI 生成的內容不具著作權**，僅人類有實質創作貢獻的部分受保護。這影響後續商業授權與使用條款設計。

---

### 3.5 🧪 測試（Testing） 🔥

五大測試類型，**每種目的不同**，考試最愛考「情境→選對測試」：

```
┌──────────────────────────────────────────────────┐
│  測試類型快速對照                                 │
├──────────┬────────────────┬─────────────────────┤
│  UAT     │ 使用者驗收     │ 真實使用者說 OK    │
│  A/B     │ 兩版比效果     │ 數據分 A 組 B 組   │
│  Beta    │ 小眾提前試用   │ 發掘早期 bug        │
│  紅隊    │ 主動攻擊找洞   │ 資安/Prompt 注入   │
│  並行運行│ 新舊同時跑     │ 對比結果才切換      │
└──────────┴────────────────┴─────────────────────┘
```

#### 3.5.1 UAT 使用者驗收測試（User Acceptance Testing）

實際使用者（不是工程師）在接近正式環境試用，**確認流程可用、符合需求**。UAT 過了才算專案交付完成。

🗣️ 就像你買家具請人組好，**你親自坐一下、確認不搖晃**——這就是 UAT。

#### 3.5.2 A/B 測試（A/B Testing）

同時讓一部分人看 A 版本、一部分人看 B 版本，**用數據比哪個效果好**（轉換率、點擊率、滿意度）。

🗣️ 就像 Instagram 發兩款封面，看哪個按讚多——**要有數據才選得動**。

#### 3.5.3 Beta 測試（Beta Testing）

小範圍真實用戶**搶先試用**，收集真實世界 bug。

🗣️ 就像 LINE 某功能先給少數用戶玩——他們幫忙抓蟲。

#### 3.5.4 紅隊測試（Red Team Testing） 🔥

扮演攻擊者角色，**主動找系統漏洞**：Prompt Injection、越獄、偏見、資料外洩。用於資安與 LLM 安全性。

🗣️ 就像學校請駭客社來試著攻擊選課系統——**先被自己人打，總比考試當天被外人打爆好**。

#### 3.5.5 並行運行（Parallel Run）

新系統跟舊系統**同時跑一段時間**，比對兩邊結果差異，確定 OK 再切換。

🗣️ 就像銀行換新系統，新舊兩套系統跑三個月——萬一新系統算錯利息，舊系統還能當對照。

#### 3.5.6 測試重點：領域正確性 + 幻覺偵測 🔥

歷屆考古題常問「醫療 GenAI 測試階段最應關注什麼？」——答案不是單純「效能」，是 **領域準確性 + 避免幻覺錯誤**。GenAI 測試跟傳統軟體不同，要把「答對率、幻覺率、事實一致性」列為 KPI。

## Section 5: 口訣 / Mnemonics

### 5.1 SMART 五字口訣：「具·量·達·關·期」

```
具 體 (Specific)
量 化 (Measurable)
達 成 (Achievable)
關 聯 (Relevant)
期 限 (Time-bound)
```

🗣️ 記法：「**具量達關期**」——唸起來像「巨量達觀期」，想像一群 AI 專案在「達觀期」看著具體數字，就記住了。

### 5.2 資源四維口訣：「人·錢·料·時」

```
👥 人 — 人力
💵 錢 — 預算
📦 料 — 資料（最常被忽略！）
⏰ 時 — 時間
```

🗣️ 「人錢料時」諧音「人前料事」——**在人前要能先料理好事情**，四樣少一樣都不行。

### 5.3 導入三步口訣：「證·試·展」

```
證 — POC 驗證（Proof of Concept）
試 — Pilot 試點
展 — Scale 擴展
```

🗣️ 「**證 試 展**」三字，就像申請工作：先**證**明能力（履歷）→**試**做一天（試用期）→正式**展**開職涯（轉正）。順序不能反。

### 5.4 因應措施四件套口訣：「HITL·備·退·查」

```
H — HITL 人類介入（高風險必備）
備 — Fallback 備援降級
退 — Rollback 回滾
查 — 幻覺偵測 + 事實查核
```

🗣️ 想像 AI 出包時的 SOP：「先叫人（HITL）→切備案（備）→退回上版（退）→查一下哪裡講錯（查）」。

### 5.5 測試五類口訣：「驗·比·試·攻·並」

```
驗 — UAT 使用者驗收
比 — A/B 測試比較
試 — Beta 小眾試用
攻 — 紅隊攻擊
並 — 並行運行
```

🗣️ 唸：「**驗比試攻並**」——像功夫招式口訣。看到情境題關鍵字（驗收、比較、試用、攻擊、並跑）就能秒對應。

## Section 6: 考試陷阱（Exam Traps）

### 陷阱 1：POC 成功就可以直接全面上線

❌ **誤解**：POC 技術跑通了，那就全公司推吧！
✅ **正解**：POC 只驗證「技術可行」，不驗證「商業價值」和「真實流程順不順」。必須經過 **Pilot 試點**（真實使用者、真實情境）才能判斷能不能 Scale。跳過 Pilot 直接 Scale 是導入失敗最常見原因之一。

---

### 陷阱 2：高風險場景可以完全自動化

❌ **誤解**：AI 這麼強，醫療診斷、貸款審核都讓 AI 自動決定最有效率。
✅ **正解**：醫療、金融、法律、勞權、公部門決策屬於**高風險場景**，必須保留 **HITL（Human-in-the-Loop）人類覆核**。**EU AI Act（強制）、NIST AI RMF（建議框架，非法規）、行政院 2023《使用生成式AI參考指引》（公部門適用）**——三者在高風險 AI 場景皆強調人類監督，但法律強制力不同。AI 可以加速建議，但最終簽名/決定必須是人。

---

### 陷阱 3：AI 導入失敗主要是技術問題

❌ **誤解**：我們 AI 選錯了、模型不夠強，所以失敗。
✅ 正解：Gartner 2025 觀察指出，GenAI 專案失敗主因涵蓋多重因素——**變革管理不足、資料品質不佳、期待落差、ROI 不明確**等非純技術因素；單一原因歸類是錯的。考生常因題材技術化就誤選「模型能力不足」，但失敗多發生在採用、流程與協作層面。

---

### 陷阱 4：目標有數字就算符合 SMART

❌ **誤解**：我們的目標是「三個月內提升客戶滿意度」——有時間有指標應該算 SMART 吧？
✅ **正解**：SMART 需**五項全部符合**。「提升客戶滿意度」缺 Measurable（沒有量化數字：從幾分到幾分？）也未必 Specific（哪個客群？哪個流程？）。正確寫法：「2026 Q3 底前，客服首次回應時間從 5 分鐘降至 2 分鐘，客訴率下降 20%。」

---

### 陷阱 5：測試類型可互相替代

❌ **誤解**：我們做過 Beta 測試就不用 UAT 了吧？都是給使用者測。
✅ **正解**：**五種測試目的不同，不能替代**：
- **UAT**：使用者驗收「需求是否達成」（上線前關卡）。
- **Beta**：小眾試用「抓真實世界 bug」（早期發布）。
- **A/B**：兩版本「比效果」（優化用）。
- **紅隊**：「主動找漏洞」（資安）。
- **並行運行**：「新舊比對、降切換風險」（高風險場景）。
題目給情境關鍵字，要選「目的對得上」的那個。

---

### 陷阱 6：並行運行等於完全替換

❌ **誤解**：反正最後都會用新系統，並行運行跟完全替換差不多嘛。
✅ **正解**：**並行運行保留舊系統作為 fallback/對照**，新系統錯了可以比對、可以切回舊系統。**完全替換是一次性下線舊系統**，壞了沒退路。在金融、醫療、法遵場景，並行運行是降低風險的標準作法。

---

### 陷阱 7：AI 專案只需要 IT 部門負責

❌ **誤解**：AI 是技術的事，交給 IT 就好。
✅ **正解**：AI 導入**必須跨部門協作**——業務提需求、法遵看合規、資安看風險、HR 做訓練、經營層對齊策略。只給 IT 做，通常會做出技術很強但業務不用、法遵違規的東西。

---

### 陷阱 8：資料夠不夠不是導入規劃要考慮的

❌ **誤解**：資料的問題讓 IT 去處理就好。
✅ **正解**：**資料是資源分配四維度之一，且是最常被低估的**。台灣僅 5–10% 企業有可用於 AI 的資料（AIF 2025）。在導入規劃階段就要盤點資料品質、數量、隱私、標註狀況。資料不足時的正解是「補資料 / 縮小範圍 / 暫緩導入」，**不是「先上線再說」**。

---

## Section 7: 情境題快速判斷

### 🔑 看到關鍵字 → 選這個答案

```
──── 目標 / KPI 類 ────
・提升滿意度（沒數字）          → 不符合 SMART
・回應時間從 5min → 2min 3 個月內 → 符合 SMART
・可量化 KPI                    → 具體數字（回應時間、錯誤率、滿意度、處理量）

──── 資源類 ────
・盤點四維資源                  → 人、錢、料、時
・最常被低估的資源              → 資料（Data）
・台灣企業資料準備率            → 僅 5–10%（AIF 2025）
・AI 導入誰負責                 → 跨部門（IT + 業務 + 法遵 + 資安 + HR）

──── 導入策略類 ────
・小規模驗證技術可行性          → POC（概念驗證）
・1–2 部門試點、真實使用者       → Pilot
・全面推展、建立 SOP             → Scale
・POC 成功可以全面上線嗎？       → 不行，要先做 Pilot
・既有系統保留作備援            → 並行運行（Parallel Run）
・新舊系統比對再切換            → 並行運行
・導入初期最常見失敗原因        → 期待落差 / 資料不足 / 跳過 Pilot
・進下一階段前要確認什麼        → 退出準則（Exit Criteria）

──── 因應措施類 ────
・高風險場景（醫療/金融/法律/勞權）→ HITL / 人類介入
・AI 最終決定誰做                → 人，不是 AI
・AI 掛了怎麼辦                  → Fallback 備援降級（切回規則或人工）
・新版上線後效果變差            → Rollback 回滾到穩定版本
・AI 講假話 / 一本正經胡說      → 幻覺偵測 + 事實查核 + HITL
・公部門使用 GenAI 要求          → 標示 AI 生成 + 人工覆核（行政院 2023 指引）
・持續監控 drift                 → AI 上線不是結束

──── 測試類型類 ────
・終端使用者驗收需求            → UAT（User Acceptance Test）
・兩版本比較效果                → A/B 測試
・小眾真實使用者提前試用        → Beta 測試
・主動攻擊找漏洞                → 紅隊測試（Red Team）
・Prompt Injection / 越獄 防範   → 紅隊測試
・醫療 GenAI 測試最應關注        → 領域正確性 + 幻覺偵測（不是純效能）
・新舊系統同時跑                → 並行運行（Parallel Run）

──── 失敗原因類 ────
・AI 導入失敗主因                → 變革管理（Change Management），不是技術
・「set it, forget it」心態       → 會導致失敗，需持續監控迭代

──── 界線類（L12301 vs L12302）────
・Build vs Buy vs SaaS           → L12301 評估（不是本課）
・成本效益分析                   → L12301 評估（不是本課）
・SMART / POC / HITL / UAT       → L12302 規劃（本課）
```

### 🎯 必考題型模板

**題型 1：SMART 判斷**
> 題目：下列哪個 AI 專案目標最符合 SMART？
> 解法：挑有「**具體場景 + 數字 + 期限**」的選項，淘汰「提升、改善、強化」等模糊用詞。

**題型 2：POC/Pilot/Scale 階段判斷**
> 題目：某公司已在研發部完成小範圍技術驗證，下一步應？
> 解法：POC 完成 → 進 **Pilot（試點）**，不是 Scale。

**題型 3：HITL 高風險場景**
> 題目：醫療機構導入 GenAI 撰寫病歷摘要，應採用何種控制？
> 解法：有「醫療/金融/法律/勞權/問責」→ 選 **HITL / 人工覆核**。

**題型 4：測試類型選擇**
> 題目：欲比較兩種 Prompt 對客服回覆品質的影響，最適合？
> 解法：「兩版本比較」→ **A/B 測試**。

**題型 5：失敗原因**
> 題目：GenAI 專案最常見的失敗原因？
> 解法：不要選「技術不成熟」，選 **變革管理 / 使用者接受度 / 期待落差 / 資料不足**。

## 結語

L12302 的核心是把「要做 AI」這個決定，**轉成一個能真的動起來、不會炸掉、出事能救火的計畫**。

記住五個關鍵字的順序：**目標 → 資源 → 策略 → 因應 → 測試**。

再記住五個口訣：
- 目標：**具量達關期**（SMART）
- 資源：**人錢料時**（四維度）
- 策略：**證試展**（POC→Pilot→Scale）
- 因應：**HITL·備·退·查**（四件套）
- 測試：**驗比試攻並**（UAT/AB/Beta/紅隊/並行）
