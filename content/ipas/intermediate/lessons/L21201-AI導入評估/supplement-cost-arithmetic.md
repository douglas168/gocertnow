|---|---|---|---|

### A.2 月度成本套算公式

```
月度 input cost (USD)  = (每日 N 次對話) × (input tokens / 次) × 30 / 1,000,000 × input price
月度 output cost (USD) = (每日 N 次對話) × (output tokens / 次) × 30 / 1,000,000 × output price
月度總 cost (USD)      = input cost + output cost
```

### A.3 範例：客服每日 1,000 次對話 / 每次 input 800 + output 400 tokens

| 模型 | input cost | output cost | 月度總（USD） | 月度總（NTD, 1USD≈31）|
|---|---|---|---|---|
| GPT-4o | $60.00 | $120.00 | **$180.00** | ~NT$5,580 |
| GPT-4o-mini | $3.60 | $7.20 | **$10.80** | ~NT$335 |
| Claude Sonnet 4.6 | $72.00 | $180.00 | **$252.00** | ~NT$7,810 |
| Claude Haiku 4.5 | $24.00 | $60.00 | **$84.00** | ~NT$2,604 |
| Gemini 2.5 Pro | $24.00 | $120.00 | **$144.00** | ~NT$4,464 |

> 同需求量下，GPT-4o-mini 比 Claude Sonnet 便宜 **~23 倍**。**任務不需要旗艦模型時，「降階」是最有效的成本控制**。

### A.4 額外成本控制工具

| 工具 | 通常折扣 | 適用情境 |
|---|---|---|
| **Batch API** | ~50% off | 非即時任務（夜間批次、資料清理） |
| **Prompt Caching** | 命中時 cached input ~90% off | 共用 system prompt、長文件 RAG |
| **降階（GPT-4o → 4o-mini）** | 5-25× 便宜 | 任務可接受品質微降 |
| **量化 / 蒸餾自建** | 視部署 | 流量穩定、長期 |

> 🔥 考題：「列出兩個降低 LLM 推論成本的做法」→ **Batch API、Prompt Caching、降階、量化**任選兩個。

## B · Break-even（損益兩平）完整算法

### B.1 通用公式

```
固定成本（每月，自建）   = F
變動成本（每萬請求，API） = V
請求量臨界點           = F / V （單位：萬請求 / 月）
```

- **量 < F/V**：用 API 划算
- **量 > F/V**：自建划算

### B.2 完整範例

**情境**：某電商計畫做商品描述自動生成。

| 項目 | 自建（TAIDE on hicloud GPU 切片） | API（GPT-4o-mini） |
|---|---|---|
| GPU 月租（1/2 切片，2025-08 hicloud 牌價示範） | NT$50,000 | — |
| MLOps 工程師時間（0.2 FTE × 月薪 NT$120,000） | NT$24,000 | — |
| 維運雜支（電費分攤、監控、模型更新） | NT$6,000 | — |
| **每月固定成本** | **NT$80,000** | NT$0 |
| 每萬請求變動成本 | ~NT$0（已含在固定） | NT$1,500 |

**損益兩平點**：

```
NT$80,000 / NT$1,500 = 53.33 萬請求 / 月
```

### B.3 圖示

```
成本（NT$/月）
     ↑
 200K│                          ／API（slope = 1,500/萬）
     │                        ／
 160K│                      ／
     │                    ／  ← 交叉點 ≈ 53 萬請求/月
 120K│                  ／
     │                ／
  80K│──────●●●●●●●●●●─────────── 自建（固定 NT$80,000）
     │     ／
  40K│   ／
     │ ／
   0 └────────────────────────────→ 月請求量（萬）
     0    20    40    60    80   100
```

### B.4 解讀

| 月請求量 | 推薦 | 備註 |
|---|---|---|
| < 30 萬 | API | 自建養 GPU 浪費 |
| 30-50 萬 | 邊緣，看趨勢 | 若預期成長快 → 開始試自建 PoC |
| 50-80 萬 | 已過交叉點 | 自建有利，但要有 MLOps 能量 |
| > 80 萬 | 強烈建議自建 | 規模效益顯著 |

> 🔥 考題若給「固定 NT$X、變動 NT$Y/萬」，要會算 X/Y。**通用公式：固定成本 ÷ (API 單價 − 自建單位變動成本)**。

## C · 加權評分矩陣完整模板與敏感度分析

### C.1 空白模板（可複製到 Excel / Notion）

| 準則 | 權重（合計 100%） | 方案 A | 方案 B | 方案 C |
|---|---|---|---|---|
| 準則 1 | __% | __ | __ | __ |
| 準則 2 | __% | __ | __ | __ |
| 準則 3 | __% | __ | __ | __ |
| 準則 4 | __% | __ | __ | __ |
| 準則 5 | __% | __ | __ | __ |
| **加權總分** | **100%** | **=Σ wᵢ×Aᵢ** | **=Σ wᵢ×Bᵢ** | **=Σ wᵢ×Cᵢ** |

### C.2 完整範例（接續 study-guide §3.6 客服 chatbot 選型）

| 準則 | 權重 | A. GPT-4o-mini | B. TAIDE 自建 | C. Claude Haiku |
|---|---|---|---|---|
| 技術效能 | 25% | 9 | 6 | 8 |
| 整合成本 | 15% | 9 | 4 | 8 |
| TCO | 30% | 9 | 5 | 7 |
| 資料主權 | 20% | 4 | 10 | 4 |
| 供應商風險 | 10% | 5 | 9 | 6 |
| **加權總分** | 100% | **7.60** | **6.50** | **6.70** |

→ **A 勝出（7.60）**。

### C.3 敏感度分析（Sensitivity Analysis）

#### 情境 1：基準（如上）

| 方案 | 加權總分 | 排名 |
|---|---|---|
| A. GPT-4o-mini | 7.60 | 1️⃣ |
| C. Claude Haiku | 6.70 | 2️⃣ |
| B. TAIDE 自建 | 6.50 | 3️⃣ |

#### 情境 2：客戶有金融屬性，資料主權權重上調

調整：資料主權 20% → 40%；TCO 30% → 10%（總和維持 100%）

| 方案 | 計算 | 加權總分 | 排名 |
|---|---|---|---|
| A. GPT-4o-mini | 0.25×9 + 0.15×9 + 0.10×9 + 0.40×4 + 0.10×5 = 2.25+1.35+0.90+1.60+0.50 | **6.60** | 2️⃣ |
| B. TAIDE 自建 | 0.25×6 + 0.15×4 + 0.10×5 + 0.40×10 + 0.10×9 = 1.50+0.60+0.50+4.00+0.90 | **7.50** | **1️⃣** |
| C. Claude Haiku | 0.25×8 + 0.15×8 + 0.10×7 + 0.40×4 + 0.10×6 = 2.00+1.20+0.70+1.60+0.60 | **6.10** | 3️⃣ |

→ **排序翻盤**：B. TAIDE 自建從第 3 變第 1。

#### 解讀

- 排序對權重「敏感」→ 代表這個決策**取決於商業情境設定**，不是純技術問題。
- 如果你不確定哪個情境是真的，**不該倉促下決定**——先去問業務、合規、客戶。
- 寫進評估報告的不只是「我們選 A」，還要寫「**在哪些假設下 A 是對的、哪些假設下會翻**」。

> 🔥 **這就是為什麼權重要先訂、且要團隊書面共識**——避免每次有人情緒上來就調權重，整份矩陣失去公信力。

## D · 雙約束過濾完整對照（延伸 §3.7）

### D.1 不同延遲 SLA 下的可行候選

| 延遲約束 | 通過的方案 | 評論 |
|---|---|---|
| p95 < **2,000 ms**（人類對話可接受） | 全部主流 API + Groq + self-host | 約束太鬆，無篩選力 |
| p95 < **1,000 ms** | Claude Haiku、Groq、self-host | 排除 Gemini Pro、GPT-4o |
| p95 < **500 ms** | Groq、self-host（部分） | 排除所有主流 frontier API |
| p95 < **200 ms**（boundary 範例） | Groq、self-host（需優化） | 只剩 specialty inference |
| p95 < **100 ms** | 多半要 edge 部署或專用硬體 | 已超出一般 LLM 場景 |

### D.2 不同 cost 約束（USD/1M input tokens）

| Cost 約束 | 通過 | 評論 |
|---|---|---|
| < $5.00 | 全部主流 API | 無篩選 |
| < $2.00 | GPT-4o-mini、Claude Haiku、Gemini 2.5 Pro、Gemini Flash | 排除 GPT-4o、Claude Sonnet |
| < $1.00 | GPT-4o-mini、Gemini Flash | 排除中階 |
| < $0.20 | GPT-4o-mini、Gemini Flash | 只剩 mini/flash 級 |

### D.3 雙約束組合：latency × cost

| latency \ cost | < $0.20/1M | < $1.00/1M | < $2.00/1M |
|---|---|---|---|
| < 200ms | self-host TAIDE | Groq、self-host | Groq、self-host |
| < 500ms | GPT-4o-mini? *| Groq、Haiku（接近） | Groq、Haiku |
| < 1000ms | GPT-4o-mini | Haiku、Gemini Pro | 多家可選 |

*GPT-4o-mini 視時段、區域可達 ~500-900ms p95；單獨評估時要實測。

> 🔥🔥 **重要 insight**：約束越嚴，候選越少。當約束嚴到「全部主流 API 都過不了」，你的選項只剩 **(1) specialty provider（Groq/SambaNova）、(2) self-host、(3) 放寬約束重新和業務談**。第三條路常常被忽略，但**有時是最理性的答案**——例如把 200ms 放寬到 500ms 後，可選方案多 5 倍、成本降一半。

## E · 整合速查：什麼數字寫進提案？

寫一份 AI 導入提案，最後要呈給老闆/董事會看的數字通常只有 4 個：

1. **TCO**（一年總投入）：「本案一年總成本 NT$X 萬」
2. **ROI**（百分比）：「預估 ROI 150%」
3. **Payback Period**（年/月）：「約 5 個月回本」
4. **加權評分排序**：「在 5 個準則的加權評分中，方案 A 以 7.6 分勝出」

附上：
- TCO 三類成本明細（**直·間·機**）
- 雙約束過濾過程（誰被淘汰、為什麼）
- 加權評分敏感度分析（哪個假設變動會翻盤）
- PoC 計畫（時程、樣本量、成功指標——pre-defined）

> L21201 的「最終交付物」就是這份提案。下一步「執行計畫、RACI、5 階段落地」是 **L21202 規劃** 的範疇。
